Project description

The main motivation for this project is to have real-life-like IT metrics to be able to do feed our R&D and analytics initiatives with meaningful data.

The project aims to create a large scale database of annotated machine metrics in the form of time series. This database, will later be used in a PhD project to test deep learning methods on. The metrics will be an output of a discrete event simulation model of a realistic system with a certain workload, such as log-ins or incoming transactions to be processed, and topology of servers communicating with each other. Independent failures and cascading failures will be simulated in the system to see the effect on the machine metrics, later to be annotated as anomalies.

Discrete event simulation

A discrete event simulation is modelled using the Simpy library.

The servers are modelled as Simpy pre-emptive resources which specifies that if a server is handling a first request and then a higher priority request comes to the server, the priority request will replace the old request. These requests are themselves handled as Simpy processes. The time between the different requests is determined by a seasonality file and a max volume. Timescale of the simulation is in seconds. The various errors are generated by blocking the server resource for a certain duration which would results in an overload of the server capacity. These errors follow a random pattern which is based on specified parameters. To model a realistic system, a network of different user specified servers is created. The connections between the servers can take different kind of routes and can be multiple for each server. Different routes through the different kinds of servers is possible.

Error Specification

The simulation takes various kinds of errors as input to model different relevant scenarios

The errors were introduced in the system in a number of ways. The first type of error modelled is when the servers reach either CPU Usage overload or Memory Usage overload. When the servers reach 100% capacity the servers will crash and no further requests shall be added. A second type of error results from the connections between servers being severed. Without communications, the network cannot work further. Thirdly, an infinite loop could occur where the simulation never ends unless an external intervention occurs. Finally, a faulty patch could make the anomalies be processed within 10 seconds at the place of 1 seconds before update. In order to see the end of the incidence, the system also records when the anomaly was solved and in what way.
